# PLAN

## Задача

Собрать в этом репозитории демонстрационный проект на `PHP` и `Laravel`, который показывает подход EDD через события:

- без использования инфраструктуры исходного проекта;
- без переноса его реальных сущностей, данных и терминов;
- с нейтральным примером, где акцент на принципе, а не на конкретном бизнес-контексте.

Проект должен отвечать на три вопроса:

1. где находится доменная логика;
2. как команда приводит к появлению доменного события;
3. как Laravel используется как оболочка, но не подменяет собой домен.

## Почему Laravel

Laravel здесь нужен не как источник бизнес-логики, а как удобный каркас:

- контейнер зависимостей;
- консольные команды;
- структура приложения;
- простая точка входа для demo-сценария;
- возможность позже подключить HTTP, queue, storage, если это понадобится.

Но в демонстрации важно сохранить границы:

- домен не должен зависеть от Eloquent;
- агрегаты не должны знать про Laravel events;
- application layer не должен быть размазан по контроллерам и моделям;
- инфраструктура Laravel должна лишь исполнять сценарии и связывать зависимости.

## Демонстрационный обезличенный домен

Чтобы пример был максимально понятным, берем не абстрактный `Request`, а сущность `Order`.

Это хороший учебный бизнесовый сценарий, потому что почти любому разработчику и аналитику интуитивно понятен жизненный цикл заказа:

- заказ создается;
- заказ подтверждается;
- заказ может быть отменен;
- заказ завершается.

Такой поток хорошо подходит для демонстрации EDD, потому что в нем есть:

- очевидный aggregate root;
- понятные инварианты;
- естественные доменные события;
- подписчики на события;
- read-side проекции.

## Сущности примера

### Aggregate Root

- `Order`

Это основной агрегат, который контролирует все переходы состояний и публикует доменные события.

### Value Objects

- `OrderId`
- `CustomerId`
- `ItemId`
- `Quantity`

Value objects нужны, чтобы пример был ближе к реальному доменному моделированию и не сводился к примитивам вроде "строка везде".

### Статусы

- `DRAFT`
- `CONFIRMED`
- `CANCELLED`
- `FULFILLED`

### Доменные события

- `OrderCreated`
- `OrderConfirmed`
- `OrderCancelled`
- `OrderFulfilled`

### Команды / use cases

- `CreateOrder`
- `ConfirmOrder`
- `CancelOrder`
- `FulfillOrder`

## Что именно хотим показать

### Базовый принцип

Команда выражает намерение.
Агрегат проверяет бизнес-правила.
Если переход допустим, агрегат меняет свое состояние и записывает доменное событие.
После сохранения агрегата события публикуются в обработчики.
Обработчики обновляют read-model или запускают побочные реакции.

### Ключевая мысль

EDD в этом примере не равно "Laravel Event::dispatch".

Нужно показать разницу между:

- доменными событиями как частью модели;
- Laravel-механикой доставки/обработки этих событий;
- инфраструктурой хранения и чтения.

## Бизнес-правила примера

Чтобы сценарий был наглядным, фиксируем простые инварианты:

- `ConfirmOrder` допустим только из состояния `DRAFT`;
- `CancelOrder` допустим только из состояния `DRAFT` или `CONFIRMED`;
- `FulfillOrder` допустим только из состояния `CONFIRMED`;
- после `CANCELLED` или `FULFILLED` заказ менять нельзя.

Это позволит ясно показать, что:

- инварианты принадлежат агрегату;
- event handlers не должны решать, можно ли менять состояние;
- событие описывает уже свершившийся факт.

## Архитектурная схема

Проект делим на 4 логических слоя.

### 1. Domain

Содержит:

- агрегаты;
- value objects;
- доменные события;
- доменные исключения;
- интерфейсы репозиториев;
- базовый `AggregateRoot` с накоплением `recordedEvents`.

Не содержит:

- Eloquent models;
- facades Laravel;
- DB query builder;
- HTTP request/response;
- queue/job кода;
- фреймворк-специфичных атрибутов и трейтов.

### 2. Application

Содержит:

- handlers / use cases;
- orchestration логики;
- загрузку агрегата;
- вызов методов агрегата;
- сохранение через repository;
- публикацию доменных событий через абстракцию event bus.

Этот слой координирует сценарий, но не держит бизнес-правила внутри себя.

### 3. Projections / Read Model

Содержит:

- обработчики доменных событий;
- модель чтения для списка заказов;
- модель чтения для карточки заказа;
- отдельные read-store классы.

Идея: write-side меняется через агрегат, а read-side обновляется реакцией на события.

### 4. Infrastructure / Laravel

Содержит:

- реализации repository;
- реализацию event bus;
- service providers;
- artisan command для demo;
- bootstrap wiring.

На первом этапе инфраструктура будет in-memory либо максимально простой файловой.

## Как это встроить в Laravel

Нужно использовать Laravel аккуратно, без смешивания слоев.

### Предлагаемая структура

```text
app/
  Domain/
    Shared/
      DomainEvent.php
      AggregateRoot.php
    Order/
      Order.php
      OrderStatus.php
      OrderRepository.php
      Events/
        OrderCreated.php
        OrderConfirmed.php
        OrderCancelled.php
        OrderFulfilled.php
      ValueObjects/
        OrderId.php
        CustomerId.php
        ItemId.php
        Quantity.php
      Exceptions/
        InvalidOrderTransition.php

  Application/
    Order/
      Commands/
        CreateOrder.php
        ConfirmOrder.php
        CancelOrder.php
        FulfillOrder.php
      Handlers/
        CreateOrderHandler.php
        ConfirmOrderHandler.php
        CancelOrderHandler.php
        FulfillOrderHandler.php
      EventBus.php

  Projections/
    Order/
      OrderDetailsProjector.php
      OrderListProjector.php
      OrderViewStore.php

  Infrastructure/
    Order/
      InMemoryOrderRepository.php
    Events/
      LaravelDomainEventBus.php
    Projections/
      InMemoryOrderViewStore.php

  Console/
    Commands/
      RunOrderDemo.php

bootstrap/app.php
routes/console.php
```

Если для лаконичности часть классов будет удобнее сгруппировать иначе, это допустимо, но смысловые границы нужно сохранить.

## Варианты технической реализации

### Вариант 1. Максимально простой старт

- Laravel как каркас;
- in-memory repository;
- in-memory read store;
- artisan command запускает сценарий;
- результат выводится в консоль.

Это лучший старт для обучения, потому что не отвлекает на миграции и БД.

### Вариант 2. Следующий шаг после базы

- добавить Eloquent-реализацию repository;
- read-model хранить в SQLite;
- часть обработчиков событий сделать queued.

Этот вариант не нужен в первом проходе, но можно оставить как развитие.

## Декомпозиция задачи

Ниже задача разложена на конкретные блоки, чтобы реализацию можно было вести последовательно и без смешения слоев.

### Блок 1. Подготовка каркаса проекта

Цель:
получить чистый Laravel-проект, в котором можно безопасно строить демонстрационный пример.

Подзадачи:

- развернуть новый Laravel-каркас в текущем репозитории;
- убедиться, что `php artisan` запускается;
- привести структуру к удобному виду для EDD-примера;
- оставить только тот framework-каркас, который реально нужен для демо.

Результат блока:

- проект запускается локально;
- есть базовый Laravel bootstrap;
- есть место для слоев `Domain`, `Application`, `Projections`, `Infrastructure`.

### Блок 2. Формализация домена

Цель:
зафиксировать бизнесовый язык примера до начала кодирования.

Подзадачи:

- утвердить aggregate root `Order`;
- утвердить value objects;
- утвердить статусы заказа;
- утвердить список команд;
- утвердить список доменных событий;
- утвердить набор инвариантов.

Результат блока:

- у команды реализации нет двусмысленности в названиях и переходах состояний;
- доменная модель понятна без чтения инфраструктурного кода.

### Блок 3. Базовые доменные абстракции

Цель:
подготовить фундамент, на котором будет строиться агрегат.

Подзадачи:

- создать интерфейс `DomainEvent`;
- создать базовый `AggregateRoot`;
- определить механизм накопления `recordedEvents`;
- определить контракт репозитория агрегата;
- определить формат доменных ошибок.

Результат блока:

- есть минимальный общий слой для всех доменных объектов;
- агрегат может записывать события и отдавать их наружу после выполнения команды.

### Блок 4. Реализация агрегата `Order`

Цель:
вынести все бизнес-правила в один понятный aggregate root.

Подзадачи:

- реализовать фабричный метод создания заказа;
- реализовать методы `confirm`, `cancel`, `fulfill`;
- встроить проверки допустимых переходов;
- публиковать правильные доменные события при каждом переходе;
- запретить недопустимые действия через domain exception.

Результат блока:

- вся основная бизнес-логика сосредоточена в агрегате;
- по коду агрегата видно, как EDD управляет переходами состояний.

### Блок 5. Application layer

Цель:
оформить команды как отдельные use cases, координирующие работу домена.

Подзадачи:

- создать command DTO/objects;
- создать handler для каждой команды;
- реализовать шаблон выполнения команды;
- разделить обязанности между handler и aggregate;
- встроить публикацию доменных событий после сохранения агрегата.

Результат блока:

- команда не содержит бизнес-правила;
- handler оркестрирует сценарий;
- агрегат отвечает за правила и события.

### Блок 6. Event bus и публикация событий

Цель:
сделать понятную схему доставки доменных событий в проекторы и реакции.

Подзадачи:

- определить интерфейс `EventBus`;
- реализовать адаптер публикации для Laravel;
- решить, как регистрируются подписчики;
- сделать публикацию списка `recordedEvents`;
- отделить domain event от framework event-механики.

Результат блока:

- события публикуются после изменения агрегата;
- инфраструктура доставки не просачивается в доменную модель.

### Блок 7. Read-side и projection layer

Цель:
показать ключевую идею EDD о разделении write-side и read-side.

Подзадачи:

- определить read-model для карточки заказа;
- определить read-model для списка заказов;
- реализовать store для проекций;
- реализовать projectors, реагирующие на доменные события;
- показать обновление представлений только через события.

Результат блока:

- можно увидеть отдельную модель чтения;
- становится наглядно, что агрегат не обновляет read-model напрямую.

### Блок 8. In-memory инфраструктура

Цель:
дать проекту рабочую среду без БД и без внешних сервисов.

Подзадачи:

- реализовать `InMemoryOrderRepository`;
- реализовать `InMemoryOrderViewStore`;
- подключить все зависимости через Laravel container;
- обеспечить простой запуск demo без миграций и очередей.

Результат блока:

- проект можно запускать сразу после установки;
- инфраструктура не отвлекает от самой EDD-модели.

### Блок 9. Демонстрационный сценарий

Цель:
сделать воспроизводимый поток, который показывает всю цепочку от команды до проекции.

Подзадачи:

- создать artisan command `demo:order-flow`;
- выполнить сценарий `create -> confirm -> fulfill`;
- отдельно показать альтернативный сценарий `create -> cancel`;
- вывести историю событий;
- вывести итог агрегата;
- вывести состояние read-model.

Результат блока:

- проект можно продемонстрировать одной командой;
- весь смысл EDD виден без чтения всех файлов подряд.

### Блок 10. Документация

Цель:
объяснить идею проекта так, чтобы его можно было использовать как учебный пример.

Подзадачи:

- описать структуру проекта;
- объяснить роль каждого слоя;
- показать разницу между command, event, aggregate, projector;
- дать инструкцию запуска;
- кратко описать ограничения примера.

Результат блока:

- новый разработчик понимает архитектуру без устного пояснения;
- проект можно использовать как reference/example.

## Шаги реализации

Ниже уже не просто блоки, а практический порядок выполнения работ.

### Этап 1. Создать пустой Laravel-проект

- поднять чистый каркас Laravel в этом репозитории;
- настроить минимальный запуск;
- удалить лишний шум, не нужный для демо, если он будет мешать чтению.

Примечание:
сейчас `composer create-project` в корень не проходит, потому что директория уже не пустая.
Когда перейдем к созданию проекта, нужно будет либо:

- развернуть Laravel в текущем каталоге после очистки/переноса служебных файлов;
- либо создать временный проект и аккуратно перенести каркас сюда.

Definition of done:

- `php artisan list` работает;
- проект стартует без дополнительных сервисов;
- структура репозитория готова для следующего этапа.

### Этап 2. Описать базовые доменные контракты

- `DomainEvent`;
- `AggregateRoot`;
- контракт `EventBus`;
- контракт `OrderRepository`.

Definition of done:

- есть минимальный набор абстракций;
- названия и сигнатуры уже позволяют писать агрегат без переделки основы.

### Этап 3. Реализовать агрегат

- `Order`;
- инварианты переходов;
- накопление доменных событий;
- методы `create`, `confirm`, `cancel`, `fulfill`.

Definition of done:

- агрегат полностью покрывает жизненный цикл заказа;
- при каждом валидном переходе создается корректное доменное событие;
- недопустимые переходы приводят к domain exception.

### Этап 4. Реализовать application layer

- команды;
- handlers;
- единый шаблон выполнения:
  - загрузить агрегат;
  - выполнить доменное действие;
  - сохранить;
  - опубликовать recorded events.

Definition of done:

- каждая команда имеет отдельный handler;
- orchestration читается отдельно от бизнес-правил;
- handlers не дублируют логику агрегата.

### Этап 5. Реализовать проекции

- `OrderDetailsProjector`;
- `OrderListProjector`;
- `OrderViewStore`.

Нужно показать, что read-side строится как реакция на события, а не как прямая запись из use case.

Definition of done:

- после публикации событий обновляется read-model;
- данные для чтения можно получить независимо от агрегата;
- проекции не вмешиваются в write-side логику.

### Этап 6. Реализовать Laravel-инфраструктуру

- привязки интерфейсов в service provider;
- `InMemoryOrderRepository`;
- `LaravelDomainEventBus`;
- `InMemoryOrderViewStore`.

Definition of done:

- контейнер собирает все зависимости;
- demo-сценарий не требует реальной БД;
- все зависимости заменяемы при дальнейшей эволюции проекта.

### Этап 7. Сделать demo-сценарий

- artisan command, например `php artisan demo:order-flow`;
- сценарий создаст заказ, проведет допустимые переходы и выведет:
  - события;
  - текущее состояние агрегата;
  - состояние projection.

Definition of done:

- одна команда воспроизводит рабочий сценарий;
- вывод в консоль наглядно показывает цепочку command -> aggregate -> event -> projection.

### Этап 8. Документация

- `README.md`;
- краткое объяснение EDD;
- как запускать demo;
- где в коде write-side, read-side, domain events и infrastructure.

Definition of done:

- проект можно понять без отдельного созвона или устного walkthrough;
- README объясняет не только "как запустить", но и "зачем так устроено".

## MVP первой версии

Чтобы не расползтись по объему, фиксируем минимальный обязательный объем первой рабочей версии.

В MVP входит:

- один агрегат `Order`;
- четыре команды;
- четыре доменных события;
- один in-memory repository;
- один in-memory read store;
- два projector-класса;
- одна artisan demo-command;
- README с объяснением потока.

В MVP не входит:

- HTTP API;
- Eloquent persistence;
- очереди;
- real broker;
- тесты на интеграцию с внешними сервисами.

## Что сознательно не делаем сначала

Чтобы проект остался демонстрационным, не включаем в первую версию:

- реальный message broker;
- HTTP API;
- авторизацию;
- внешние интеграции;
- saga/process manager;
- outbox;
- event sourcing как основную модель хранения;
- сложную многомодульную структуру;
- реальную БД, если она не нужна для демонстрации принципа.

## Критерии хорошего результата

Проект считается удачным, если:

- его можно запустить локально одной-двумя командами;
- человек быстро видит, где домен, а где Laravel;
- по коду понятно, как команда превращается в событие;
- read model отделена от write model;
- названия и сущности обезличены;
- пример не тянет за собой инфраструктурный шум.

## Риски и решения

### Риск 1. Laravel "съест" архитектуру

Если начать строить все через Eloquent model, observers и facades, пример превратится в обычный Laravel CRUD.

Решение:

- держать агрегат в `app/Domain`;
- репозитории оформить интерфейсами;
- использовать Laravel только как runtime и DI.

### Риск 2. Слишком много ceremony

Если добавить слишком много абстракций, пример станет теоретическим и тяжело читаемым.

Решение:

- только минимальные интерфейсы;
- только один агрегат;
- только 1-2 projection;
- только один демонстрационный сценарий.

### Риск 3. Смешение domain event и framework event

Если не разделить эти сущности, идея EDD станет размытой.

Решение:

- отдельный интерфейс доменного события;
- отдельный адаптер публикации в Laravel;
- явное именование в коде и документации.

## Практический порядок после этого плана

Когда начнем реализацию, идти нужно так:

1. развернуть чистый Laravel-каркас в этом репозитории;
2. подготовить папки `Domain`, `Application`, `Projections`, `Infrastructure`;
3. реализовать `Order` агрегат и события;
4. подключить in-memory repository и event bus;
5. добавить projectors и read-store;
6. сделать artisan demo-command;
7. написать `README.md`.

## Ожидаемый результат первой рабочей версии

На выходе хотим получить минимальный Laravel-проект, где можно выполнить команду вида:

```bash
php artisan demo:order-flow
```

И увидеть:

- какие команды были выполнены;
- какие доменные события возникли;
- в каком состоянии агрегат после сценария;
- как обновились read-model представления.
